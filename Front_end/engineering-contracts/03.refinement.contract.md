# Refinement Contract — Post-Development

> **When to use:** After implementation is complete, before marking a task as done or opening a pull request.
> This contract is a structured audit — not an optional cleanup pass. Every section must be evaluated.

---

## 1. Responsibility Audit

For every file that was created or modified:

- [ ] Can you name its **single responsibility** in one sentence?
- [ ] Does the file contain logic that belongs to a **different layer**? (extract it)
- [ ] Does the file contain logic that belongs to a **different feature**? (move it)
- [ ] Is there any **dead code** introduced by this change? (delete it)

**Signal for a problem:** If you need more than one sentence to describe what a file does, it has mixed responsibilities.

---

## 2. Layer Integrity Audit

Walk the dependency graph for every changed file:

- [ ] Does the file import anything from a **higher layer** than itself?
- [ ] Does a **component** call an API, create an API client, or import from `app/services/` directly?
- [ ] Does a **page** contain business orchestration, async operations, or state transitions?
- [ ] Does a **controller** do anything other than wire state and callbacks?
- [ ] Does a **shared package** import React, Zustand, or any app/feature code?

If any of these are true, refactor before proceeding.

---

## 3. Import Audit

Review every `import` statement in changed files:

- [ ] No **deep package imports** (must use root barrels only)
- [ ] No **cross-feature imports** (features must not import each other's internals)
- [ ] No **cross-app imports** (apps must not import each other)
- [ ] No **orphaned imports** (every import is actually used)
- [ ] No **circular dependencies** introduced

Run a static check if possible:

```bash
# Check for unused imports (ESLint rule: no-unused-vars / unused-imports plugin)
# Check for circular dependencies
npx madge --circular src/
```

---

## 4. Type Safety Audit

Review every TypeScript declaration introduced:

- [ ] Zero `any` — replace with `unknown` + type guard if needed
- [ ] Zero unchecked casts (`as SomeType`) without preceding runtime check
- [ ] Zero implicit `undefined` paths — optional chaining (`?.`) and nullish coalescing (`??`) applied where needed
- [ ] All async states are modeled (`idle | loading | success | error`)
- [ ] All new public function signatures have explicit parameter and return types
- [ ] View models, command types, and result types are named and exported correctly

---

## 5. State Integrity Audit

- [ ] Is state **duplicated** anywhere? (same value tracked in two places)
- [ ] Does the new state have a clear **owner**? (store, component, or server cache — not ambiguous)
- [ ] Are **query keys** typed as constants and colocated with their hook?
- [ ] Are **query invalidations** triggered correctly after mutations?
- [ ] Is Zustand used only for UI/optimistic state — not for server-fetched data?
- [ ] Is there any residual state that should be **reset** on unmount or navigation?

---

## 6. Domain Mapper Audit

- [ ] Are there any **raw backend DTOs** in stores, components, or view logic?
- [ ] Does every API response pass through a **typed mapper** before entering the domain?
- [ ] Are mappers **pure functions** — no side effects, no API calls, no store mutations?
- [ ] Are mapper output types the **same domain types** used in the rest of the feature?

---

## 7. Error Handling Audit

- [ ] Is every `try/catch` surfacing errors to the UI (not just `console.error`)?
- [ ] Are errors represented as **typed values** — not raw `unknown` or `any`?
- [ ] Does every async flow handle the **loading**, **success**, and **error** states visibly in the UI?
- [ ] Are network errors and domain errors **distinguished** when they require different UI treatment?
- [ ] Is there any place where a rejected promise is **unhandled**?

---

## 8. Component Quality Audit

- [ ] Do all components that accept handlers have **typed callback props** (`onX: () => void`)?
- [ ] Are there any components that are **too large** to reason about in one reading? (split by responsibility)
- [ ] Are there any components that receive **entire domain objects** as props instead of the specific fields they need? (narrow the props)
- [ ] Are there any `key` props missing from rendered lists?
- [ ] Are there any **conditional hooks** (hooks inside `if` blocks)? (React rules of hooks violation)
- [ ] Are there any **effects with missing dependencies** in `useEffect`?

---

## 9. Performance Review

Evaluate whether the implementation introduces unnecessary rendering or computation:

| Check | Signal |
|---|---|
| Unnecessary re-renders | Component subscribes to more store state than it uses |
| Missing memoization | Expensive computed values recalculated on every render |
| Waterfall requests | Sequential API calls that could be parallel |
| Over-fetching | Fetching full collections when only IDs are needed |
| Effect-as-lifecycle | `useEffect` used for data fetching instead of TanStack Query |

**Memoization guidelines:**
- Use `useMemo` for computationally expensive derived values — not as a default
- Use `useCallback` for callback props passed to memoized children — not for every function
- Use `React.memo` for pure display components that re-render frequently
- Do not prematurely memoize — measure first

---

## 10. Bundle Impact Review

For any new dependency or large feature addition:

- [ ] Is the new dependency already available in the project? (check `package.json` before adding)
- [ ] Are large components or pages **lazy loaded** with `React.lazy()` + `Suspense`?
- [ ] Are there any imports of entire libraries where only one function is needed?

```typescript
// Forbidden — imports entire library
import _ from 'lodash';

// Required — named import only
import { groupBy } from 'lodash-es';
```

---

## 11. Accessibility Audit

- [ ] Do all interactive elements have accessible labels (`aria-label`, `aria-labelledby`, or visible text)?
- [ ] Do all images have `alt` text?
- [ ] Is keyboard navigation possible for all interactive flows?
- [ ] Is focus managed correctly after async operations (modals, drawers, alerts)?
- [ ] Are error messages associated with their inputs via `aria-describedby`?
- [ ] Is color used as the **only** differentiator for any state? (must have a secondary indicator)

---

## 12. Security Review

- [ ] No `dangerouslySetInnerHTML` without explicit sanitization
- [ ] No auth tokens, credentials, or secrets in `localStorage`, `console.log`, or component state
- [ ] No user-controlled input rendered as HTML or used to construct queries without sanitization
- [ ] All user-visible IDs validated server-side — no IDOR reliance on client-side checks

---

## 13. Architecture Documentation Sync

If the change affects project-level architecture:

- [ ] Does `AGENTS.md` still accurately describe the frontend architecture after this change?
- [ ] Was a new shared package created? Add it to the allowed imports list in `AGENTS.md`.
- [ ] Was a new feature domain introduced? Confirm it is named and structured consistently.
- [ ] Does `CLAUDE.md` need any updates to reflect new conventions introduced?

---

## 14. Final Pull Request Readiness Checklist

- [ ] All TypeScript compiles without errors: `tsc --noEmit`
- [ ] All lint rules pass with no suppressions added
- [ ] No `console.log`, `TODO`, or `FIXME` left in production code paths
- [ ] No commented-out code blocks
- [ ] No orphaned files (newly created files that nothing imports)
- [ ] No unresolved merge conflicts
- [ ] Commit messages describe the **why**, not the what
- [ ] PR description covers: what changed, why, and how to test it
- [ ] Acceptance criteria from the planning contract are verified

---

## Refinement Decision Tree

```
Is the file's responsibility nameable in one sentence?
  No → Split the file first

Does every import respect the layer graph?
  No → Refactor the import before proceeding

Is there any `any`, silent undefined, or unchecked cast?
  Yes → Fix the type before proceeding

Is there any raw DTO in a store or component?
  Yes → Add a mapper before proceeding

Is there any unhandled error path?
  Yes → Surface it to the UI before proceeding

All checks passed?
  → Ready for review
```
