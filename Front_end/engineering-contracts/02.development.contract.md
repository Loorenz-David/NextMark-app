# Development Contract — Active Implementation

> **When to use:** While writing code. This contract defines the non-negotiable standards every line of frontend code must meet before it is considered complete.
> Apply these rules continuously — not as a final checklist.

---

## 1. TypeScript Discipline

### Strictness
- No `any`. If a type is genuinely unknown, use `unknown` and narrow it with a type guard.
- No unchecked casts (`as SomeType`) without a preceding runtime check.
- No silent `undefined` propagation — handle optional fields explicitly.
- All function parameters and return types must be typed. No implicit `any` from missing annotations.
- Prefer `interface` for extendable object shapes. Prefer `type` for unions, intersections, and mapped types.

### Null Safety
```typescript
// Forbidden
const name = user.profile.name; // may throw if profile is undefined

// Required
const name = user.profile?.name ?? 'Unknown';
```

### Async State Typing
Every async operation must model all states:
```typescript
type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: string };
```

## 2. Layer Enforcement

The dependency graph is a hard constraint, not a guideline:

```
packages → app/services → features/api → features/actions → features/flows → features/controllers → pages/components
```

**Violations to catch while coding:**

| Violation | Example | Fix |
|---|---|---|
| Component calls API directly | `fetch('/orders')` in a component | Move to `features/api/` hook |
| Page contains business logic | `if (order.status === 'pending') {...}` in a page | Move to controller or flow |
| Controller creates API client | `new ApiClient()` in a `useXController` | Inject via hook from `features/api/` |
| Shared package imports React | `import { useState } from 'react'` in `packages/` | Remove — packages are framework-agnostic |
| Feature imports another feature | `import { useOrderFlow } from '../orders'` in `routes/` | Extract to shared layer or invert dependency |
| Raw DTO in store | Storing the backend response shape directly | Map through a domain mapper first |

## 3. Feature Folder Structure

Every feature must use this folder layout — even if some folders have only one file:

```
features/
  {feature-name}/
    domain/          ← pure types, guards, view models
    api/             ← TanStack Query hooks (GET) + mutation hooks (POST/PUT/DELETE)
    actions/         ← write operation orchestrators (createX.action.ts)
    flows/           ← multi-step workflow logic (x.flow.ts)
    controllers/     ← UI state + orchestration hooks (useXController)
    stores/          ← Zustand slices (x.store.ts)
    providers/       ← React context providers
    components/      ← pure display components
    pages/           ← route-level composition components
```

Do not collapse folders because the feature is small. The boundary matters more than the file count.

## 4. Naming Conventions

Every identifier must follow the established conventions — consistency is readability at scale:

| Artifact | Convention | Example |
|---|---|---|
| Controller hook | `useXController` / `x.controller.ts` | `useRouteController` |
| Action file | `createX.action.ts` / `updateX.action.ts` | `createRoute.action.ts` |
| Flow hook | `useXFlow` / `x.flow.ts` | `useAssignDriverFlow` |
| Store file | `x.store.ts` / `x.slice.ts` | `route.store.ts` |
| Selector file | `x.selector.ts` | `route.selector.ts` |
| Mapper function | `mapXToY` | `mapOrderDtoToOrder` |
| Boolean guard | `canX` / `isX` / `shouldX` | `canAssignDriver` |
| View model type | `XViewModel` | `RouteStopViewModel` |
| Command type | `XCommand` / `XResult` | `AssignDriverCommand` |
| API hook | `useXQuery` / `useXMutation` | `useRoutesQuery` |

## 5. Component Architecture

### Single Responsibility
Each component has one job. If a component renders a list **and** handles submission logic **and** manages open/closed state — split it.

### Props Design
```typescript
// Props should be narrow and typed — not generic containers
interface RouteCardProps {
  routeId: string;
  status: RouteStatus;
  driverName: string | null;
  onAssign: (routeId: string) => void;
}
// Forbidden: passing entire store slices or domain objects as props
```

### Event Boundaries
- Components emit typed callbacks (`onX`, `onXChange`) — they do not call actions directly.
- Controllers wire callbacks to actions — components stay unaware of async operations.

### Display vs. Connected
- **Display components** — receive all data via props, emit typed events, zero store access.
- **Controller-connected components** — receive a single controller hook result, no direct store access.

```typescript
// Controller-connected page component
function RouteListPage() {
  const ctrl = useRouteController();
  return <RouteList routes={ctrl.routes} onAssign={ctrl.handleAssign} />;
}

// Pure display component
function RouteList({ routes, onAssign }: RouteListProps) {
  return routes.map(r => <RouteCard key={r.id} {...r} onAssign={onAssign} />);
}
```

## 6. State Management Rules

### Zustand Stores
- One store per feature domain — never one mega-store
- State shape must be flat — avoid deeply nested mutable objects
- Mutations happen only inside store actions (`set(state => ...)`)
- No derived state in the store — use selectors

```typescript
// Correct store structure
interface RouteStore {
  routes: Route[];
  selectedRouteId: string | null;
  status: 'idle' | 'loading' | 'error';
  setRoutes: (routes: Route[]) => void;
  selectRoute: (id: string) => void;
}
```

### TanStack Query
- All server state (fetched data) lives in TanStack Query — not in Zustand
- Zustand owns UI state and optimistic state only
- Invalidate query keys on successful mutations — never manually update query cache unless optimistic update requires it
- Query keys must be colocated with their hook, typed as a constant

```typescript
export const routeKeys = {
  all: ['routes'] as const,
  byId: (id: string) => ['routes', id] as const,
};
```

## 7. Async Operation Patterns

### Action Pattern (write operations)
```typescript
// createRoute.action.ts
export async function createRouteAction(
  command: CreateRouteCommand,
  deps: CreateRouteDeps
): Promise<CreateRouteResult> {
  const dto = mapCreateRouteCommandToDto(command);
  const response = await deps.api.createRoute(dto);
  return mapRouteDtoToRoute(response);
}
```

### Error Handling
- Never swallow errors silently
- Surface errors to the UI via controller state — not via `console.error` alone
- Use typed error shapes — not raw `catch (e: any)`

```typescript
try {
  const result = await createRouteAction(command, deps);
  dispatch({ type: 'success', data: result });
} catch (error) {
  const message = error instanceof AppError ? error.message : 'An unexpected error occurred';
  dispatch({ type: 'error', error: message });
}
```

## 8. Import Discipline

### Allowed Import Directions
- Apps may import from `packages/` via root barrels only
- Features may import from `app/services/` and `packages/`
- Components may import from their feature's `domain/`, `controllers/`, and `components/`

### Forbidden Imports
```typescript
// Deep package import — forbidden
import { mapOrderDtoToOrder } from '@shared-domain/orders/mappers/orderMapper';

// Cross-feature import — forbidden
import { useDriverStore } from '../drivers/stores/driver.store';

// Cross-app import — forbidden
import { useAdminSession } from '../../admin-app/session/adminSession';

// API client in controller — forbidden
import { createApiClient } from '@shared-api';
const client = createApiClient(); // inside a useXController
```

### Barrel Imports
- Each feature exposes a barrel (`index.ts`) for external consumption
- Internal files import relatively; external consumers use the barrel

## 9. Domain Mapper Rules

Never pass raw backend DTOs into stores, components, or view logic.

Every API response must pass through a mapper before entering the frontend domain:

```typescript
// Forbidden — raw DTO in store
store.setOrder(apiResponse); // apiResponse is a backend DTO

// Required — mapped domain type in store
const order = mapOrderDtoToOrder(apiResponse);
store.setOrder(order);
```

Mappers live in `features/{feature}/domain/` or `packages/shared-domain/`.

## 10. Security Surface — Frontend

While coding, actively guard against:

| Risk | Mitigation |
|---|---|
| XSS via `dangerouslySetInnerHTML` | Never use unless explicitly sanitizing with DOMPurify or equivalent |
| Unvalidated user input in URL params | Validate and sanitize before use in queries or display |
| Exposing auth tokens in logs | Never `console.log` tokens, sessions, or credentials |
| IDOR via client-side ID manipulation | Always validate resource ownership server-side |
| Sensitive data in `localStorage` | Never store auth tokens in `localStorage` — use `httpOnly` cookies |

## 11. Testing Standards

### What to test at each layer
| Layer | Test type | What to assert |
|---|---|---|
| `packages/` | Unit | Pure function input → output |
| `features/actions/` | Unit | Command handling, mapping, error cases |
| `features/flows/` | Unit | State transitions, branching logic |
| `features/controllers/` | Integration | Hook behavior with mocked API + store |
| `components/` | Component | Renders correctly, emits correct events |
| `pages/` | E2E (optional) | Happy path user flows |

### Rules
- Do not test implementation details — test observable behavior
- Do not mock the domain layer — test it directly
- Controller tests should use a real store + mocked API, not a mocked controller

---

## 12. Commit Standards

- One commit per logical change — not per file
- Commit message describes **why**, not what:
  - `fix: prevent duplicate route creation when save is clicked twice`
  - Not: `updated routeController.ts`
- Never commit commented-out code
- Never commit `console.log` statements
- Stage files explicitly — never `git add .` in ambiguous state
