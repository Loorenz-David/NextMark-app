# Planning Contract — Pre-Development

> **When to use:** Before writing a single line of code. This contract governs how AI assistants and engineers must analyze, scope, and design a frontend feature or change.
> If any section cannot be answered, stop and resolve it with the developer before proceeding.

---

## 1. Task Decomposition

Before planning, break the request into atomic units:

- [ ] What is the **single user-facing outcome** this task delivers?
- [ ] What are the **distinct technical sub-tasks** required to achieve it?
- [ ] Can any sub-task be deferred without blocking the core outcome? (defer it)
- [ ] Does this task touch **multiple domains**? If yes, treat each domain change as a separate sub-task.

**Rule:** One task = one domain + one responsibility. If you can't name it in one sentence, split it.

---

## 2. Requirement Classification

Classify the change type before touching architecture:

| Type | Description | Example |
|---|---|---|
| `data-shape` | A new or modified domain type | New field on `Order` |
| `api-contract` | New or changed endpoint interaction | New `POST /routes` call |
| `ui-behavior` | New or changed component interaction | Button triggers a flow |
| `state-change` | New or changed store, selector, or derived state | New `routeStore` slice |
| `orchestration` | A new multi-step workflow | Create route → assign driver |
| `cross-cutting` | Affects multiple features or apps | Change to shared package |

---

## 3. Layer Placement Decision

Every piece of logic must be assigned a layer **before** implementation:

```
packages → app/services → features/api → features/actions → features/flows → features/controllers → pages/components
```

For each sub-task, answer:

- What **layer** owns this logic?
- Does an existing file at that layer already handle adjacent concerns? (extend it or add beside it — never collapse layers)
- Does this require a **new file**? Name it using the naming conventions before creating it.
- Does this touch a **shared package**? If yes, confirm it requires no React, no Zustand, no app imports.

**Placement checklist:**

- [ ] Domain types → `packages/shared-domain/`
- [ ] API types + transport wrappers → `packages/shared-api/` or `features/api/`
- [ ] Async data fetching → `features/api/` (TanStack Query hooks)
- [ ] Business write operations → `features/actions/`
- [ ] Multi-step workflows → `features/flows/`
- [ ] UI state + orchestration → `features/controllers/` (`useXController`)
- [ ] Layout + composition → `pages/`
- [ ] Pure display → `components/`

---

## 4. Dependency Impact Assessment

Map the blast radius before starting:

- [ ] List every **file that will change**
- [ ] List every **file that imports** any of those files
- [ ] Identify whether changes are **additive** (low risk) or **destructive** (breaking signatures, renaming exports)
- [ ] Flag any change that touches a **shared package** — these affect all consuming apps

If a shared package changes, confirm:
- [ ] All consumers (`admin-app`, `driver-app`, etc.) have been checked for compatibility
- [ ] No deep import paths are being introduced

---

## 5. Type System Design

Design the TypeScript types before writing logic:

- [ ] Define new domain types (interfaces, enums) with exact field names and types
- [ ] Define view models (`XViewModel`) that translate domain types to UI-safe shapes
- [ ] Define command types (`XCommand`, `XResult`) for action inputs/outputs
- [ ] Confirm no `any` will be introduced — if a type is unknown, model it as `unknown` and narrow it explicitly
- [ ] Confirm all async states are modeled: `idle | loading | success | error`

---

## 6. State Ownership Declaration

Before touching state:

- [ ] Where does this state **live**? (`app/`, `features/stores/`, local component `useState`)
- [ ] Is this state **already tracked** somewhere? (check existing stores before creating new ones)
- [ ] Is this state **shared across features**? If yes, escalate to a shared store — never duplicate state
- [ ] Is this state **ephemeral** (local UI only) or **persistent** (needs to survive navigation)?
- [ ] What triggers **invalidation** or **reset** of this state?

**Rule:** Do not create a new store for state that is already owned by an existing store. Extend, don't duplicate.

---

## 7. API Contract Definition

For any new or modified backend interaction:

- [ ] Define the **request shape** (method, path, headers, body)
- [ ] Define the **response shape** (success payload, error shape)
- [ ] Confirm the backend endpoint exists and matches the expected contract
- [ ] Map backend DTO fields to frontend domain types — no raw DTOs in stores or components
- [ ] Define the **loading, success, and error states** the UI must handle

---

## 8. Component Hierarchy Sketch

Before writing any JSX:

- [ ] Sketch the component tree (even in a comment block)
- [ ] Identify which components are **pure display** (receive props, emit events)
- [ ] Identify which components are **controller-driven** (connected to `useXController`)
- [ ] Confirm no page component will contain business orchestration
- [ ] Confirm no component will call an API directly

---

## 9. Acceptance Criteria

Define done before starting:

- [ ] What does the UI look like when the **happy path** completes?
- [ ] What does the UI show during **loading**?
- [ ] What does the UI show on **error**?
- [ ] What does the UI show when **empty** (no data)?
- [ ] Are there **edge cases** from the domain that must be handled?

---

## 10. Risk Register

Flag anything that warrants extra scrutiny:

| Risk | Mitigation |
|---|---|
| Shared package change | Audit all consumers before merging |
| Store restructure | Check for state duplication after change |
| New async flow | Confirm all async states are handled (loading/error/empty) |
| Auth-gated feature | Confirm guard is applied at the correct layer |
| Breaking type change | Check for and update all consuming call sites |

---

## Planning Sign-Off Checklist

Before proceeding to development:

- [ ] Task decomposed into atomic units
- [ ] Every sub-task assigned to a named file at the correct layer
- [ ] Type system designed with no `any`
- [ ] State ownership declared with no duplication
- [ ] Blast radius mapped and accepted
- [ ] Acceptance criteria defined for all states (loading, error, empty, success)
- [ ] Shared package impact confirmed as zero or explicitly scoped
